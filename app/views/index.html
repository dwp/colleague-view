Colleague View prototypes

The aim of the Colleague View minimum viable product (MVP) is give customers' benefit information to agents answering the telephones, so that they can answer common questions.

The users that we are focusing on for MVP are agents that work on Employment and Support Allowance (ESA).

The scope of the MVP

The scope of the MVP is to meet agents' needs for:

  • answering questions about next payment dates and amounts
  • viewing a customer's personal details
  • adding details about a call with a customer
  • viewing details about previous calls with a customer

UR 10 insights to consider for iterations

What users said about:

Filter

  • the benefit one is useful
  • I wouldn't usually search by date
  • looking for previous conversations, it's more likely we would look for other questions, not payment queries, as these would usually be answered straight away'.
  • users feel confident in completing the call or starting the service again
  • the ability to filter by type of contact was helpful, but users expressed the need to also filter based on if the contact was inbound or outbound (TCFC - acronym for telephone call from customer).
  • request to have 'letters' and 'emails' specifically called out rather than be included in 'A different type of contact' - 'Especially letters, because they are quite common'
  • it would be easier if it [the filter] was all on one screen without having to scroll' 'It's quicker if it's [all] on the screen'.
  • ucd colleagues suggested that we use drop down options for the filter to shorten it. For example, rather than listing all queries immediately, include a drop down so that users can choose to show all queries they can filter against if this is what they are wanting to filter against.
  • there is a potential to have the filter be dynamic depending on who is logged in once role based access is implemented.
  • user suggested going with the drop down option - 'If I need a payment enquiry then I can click it, but if not, I don't need to see all of the different types'.

Summary card

  • I prefer benefit first and notes last.
  • most users prefer fewer lines of notes visible straight away with 'See more' link after 2-3 lines.
  • the term 'implicit consent' is used if contact is with a third party. The agent will only give limited information to third parties. A user suggested this be included in the 'More details' section of the contact log.
  • what security the person has passed is 'good to have'.
  • 'I don't know what that means' - DLA agent on 'CHaRM referral'
  • user was worried about security and privacy - should all agents have the ability to see all contact notes for all benefits?
